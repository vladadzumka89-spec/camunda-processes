# 🏁 Start — почати роботу над задачею

Користувач хоче почати роботу над новою задачею. Аргумент `$ARGUMENTS` — опис задачі.

## Крок 1: Перевір що є опис задачі

Якщо `$ARGUMENTS` порожній — запитай через AskUserQuestion:
- "Опишіть задачу над якою будете працювати"
- Варіанти: вільний ввід

## Крок 2: Уточни задачу у користувача

Перш ніж починати — постав 2-4 уточнюючих питання через AskUserQuestion. Питання мають бути **простими, зрозумілими, без технічних термінів** — ніби ти розмовляєш зі звичайною людиною.

Мета — зрозуміти:
- **Що саме** користувач хоче отримати (конкретний результат)
- **Для кого** це робиться (хто буде цим користуватись)
- **Як це має виглядати/працювати** з точки зору кінцевого користувача
- **Чи є обмеження** або особливі побажання

Приклади хороших питань:
- "Де саме в системі це має з'являтись — у якому розділі/вікні?"
- "Хто буде цим користуватись — менеджери, бухгалтери, всі?"
- "Як ви уявляєте результат — кнопка, нова сторінка, автоматична дія?"
- "Чи має це працювати з якимись існуючими даними/документами?"
- "Чи є приклад або зразок як це має виглядати?"
- "Що має відбуватись після виконання цієї дії?"
- "Чи потрібне якесь підтвердження від користувача перед дією?"

**ЗАБОРОНЕНО** питати про:
- Назви модулів, моделей, полів
- Технічну архітектуру
- Бази даних, API, бекенд/фронтенд
- Будь-які програмістські терміни

Задавай питання одним блоком AskUserQuestion (2-4 питання за раз). Дочекайся відповідей перед продовженням.

## Крок 3: Синхронізуйся з main (безпечно, без втрати змін)

Перевір поточну гілку та стан:
```bash
git branch --show-current
git status --short
```

```bash
git fetch origin main
```

Якщо є незакомічені зміни — запитай через AskUserQuestion:
> У вас є незакомічені зміни у наступних файлах:
> (покажи список файлів)
- Варіанти: "Закомітити в поточну гілку" / "Перенести зміни в нову гілку" / "Видалити зміни" / "Скасувати"

Якщо "Закомітити в поточну гілку":
```bash
git add -A
git commit -m "WIP: auto-save before /start"
```
Повідом: "Зміни закомічено в гілку `CURRENT_BRANCH`. Можна повернутись до них пізніше."
Нова гілка буде чистою.

Якщо "Перенести зміни в нову гілку":
```bash
git stash push -m "auto-stash before /start: TASK_DESCRIPTION"
```
Зміни будуть відновлені в новій гілці після її створення (Крок 5).

Якщо "Видалити зміни":
```bash
git checkout -- . && git clean -fd
```
Повідом: "Незакомічені зміни видалено."

Якщо "Скасувати" — зупинись.

## Крок 4: Згенеруй назву гілки

На основі опису задачі (з урахуванням уточнень з кроку 2) згенеруй **описову** назву гілки.

Формат: `PREFIX/short-description`

Префікси:
- `feat/` — нова функціональність
- `fix/` — виправлення бага
- `refactor/` — рефакторинг без зміни поведінки
- `style/` — візуальні зміни, UI/UX
- `chore/` — інфраструктура, конфігурація

Правила:
- Потім через дефіс — суть змін англійською
- Тільки латинські літери, цифри та дефіси
- Максимум 60 символів після префікса
- Все у нижньому регістрі
- Назва має бути **достатньо описовою** щоб з неї було зрозуміло суть роботи

Приклади:
- "Додати нагадування в BPMN процес" → `feat/bpmn-add-reminder-timer`
- "Фікс: webhook не відправляє correlation key" → `fix/webhook-missing-correlation-key`
- "Новий DMN для класифікації рахунків" → `feat/dmn-invoice-classification`
- "Рефакторинг OdooClient на HTTP" → `refactor/odoo-client-http-webhook`

## Крок 5: Створи гілку від origin/main (без reset)

**НЕ робити `git reset --hard`** — це знищує зміни.

Перевір чи гілка з такою назвою вже існує:
```bash
git branch --list BRANCH_NAME
git ls-remote --heads origin BRANCH_NAME
```

Якщо гілка вже існує — додай суфікс `-v2`, `-v3` і т.д.

Створи гілку напряму від origin/main:
```bash
git checkout -b BRANCH_NAME origin/main
```

Якщо в Кроці 3 користувач обрав "Перенести зміни в нову гілку" — відновити stash:
```bash
git stash pop
```
Якщо stash pop дав конфлікти — повідом користувача і покажи конфліктні файли.

## Крок 6: Стартуй BPMN процес у Camunda (створює задачу в Odoo)

На основі опису задачі та уточнень сформулюй:
- `task_name` — коротка назва задачі (1 речення, зрозуміле менеджеру)
- `task_description` — детальний опис що потрібно зробити (HTML, може містити `<p>`, `<ul>`, `<li>`)

Стартуй процес `feature-to-production` через Camunda REST API:

```bash
curl -s -X POST "http://10.1.1.61:8088/v2/process-instances" \
  -H "Content-Type: application/json" \
  -u "demo:demo" \
  -d '{
    "processDefinitionId": "feature-to-production",
    "variables": {
      "task_name": "TASK_NAME",
      "task_description": "TASK_DESCRIPTION",
      "head_branch": "BRANCH_NAME"
    }
  }'
```

Перевір відповідь:
- Якщо HTTP 200 і є `processInstanceKey` — процес стартував успішно
- Якщо помилка — повідом користувача, але продовжуй роботу (задача в Odoo створюється асинхронно)

**Що відбувається:** Camunda виконує BPMN Phase 0 → Service Task створює батьківську задачу в Odoo через webhook → процес чекає на створення PR (Phase 1 стартує автоматично коли `/ship` створить PR).

## Крок 7: Залиш маркер сесії

Створи файл-маркер щоб `/ship` знав контекст роботи:
```bash
echo "ОПИС_ЗАДАЧІ" > .claude-task
```

Цей файл НЕ комітиться (він вже в `.gitignore` або буде ігноруватись).

## Крок 8: Підтвердження

Покажи користувачу:

> 🏁 **Робочу гілку створено!**
>
> - 🌿 Гілка: `BRANCH_NAME`
> - 📋 Задача: TASK_NAME
> - 📌 Базова гілка: `main`
> - ✅ Задачу створено в Odoo (через Camunda BPMN)
>
> Починаю роботу. Після завершення використай `/ship` для відправки.

## Крок 9: Почни роботу

Після створення гілки одразу починай працювати над описаною задачею. Аналізуй опис задачі та дій відповідно — досліджуй код, плануй зміни, пиши код.
