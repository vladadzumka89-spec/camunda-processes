# Виправлення BPMN-файлу під стандарти проєкту

Коли користувач надає BPMN-файл (шлях або завантажений файл), виконай наступне:

## Крок 1: Збір інформації

Перед редагуванням ОБОВ'ЯЗКОВО запитай у користувача:

1. **Який ID проєкту в Odoo?** (число з URL проєкту, наприклад `252`)
2. **Який Process ID має бути?** (латиницею, kebab-case, наприклад `oplata-rakhunkiv`)
3. **Яка назва процесу?** (українською, для відображення)
4. **Це процес для продакшн бази чи для демо?**
   - Якщо **демо** → використовуй webhook: `http://odoo.dev.dobrom.com/web/hook/21c8dbff-86e8-4005-9bfc-9f77ee9b5c57`
   - Якщо **продакшн** → використовуй webhook: `https://o.tut.ua/web/hook/8531324a-2785-48d1-8f4d-ddd66a267d50`
5. **Які ролі (swim lanes) потрібні?** (наприклад: Бухгалтер, Менеджер, Керівник)
6. **Нагадування та ескалації — додавати до всіх User Tasks чи тільки до окремих?** Якщо до окремих — до яких саме? (За стандартом кожен User Task має мати нагадування + дедлайн, але замовник може вирішити інакше)

## Крок 2: Аналіз файлу

Прочитай BPMN-файл та перевір на відповідність правилам з CLAUDE.md. Покажи користувачу список знайдених проблем у форматі:

```
АНАЛІЗ BPMN: <назва файлу>

✅ Що вже правильно:
- ...

❌ Що потрібно виправити:
1. [Критично] Немає XOR gateway для перевірки odoo_task_id
2. [Критично] User Tasks мають <zeebe:userTask /> без ioMapping
3. [Важливо] Немає swim lanes
4. [Важливо] Boundary events відсутні на User Task "..."
5. [Рекомендація] Process ID "Process_abc123" — краще замінити на зрозумілий

💡 Пропозиції покращення:
- ...
```

## Крок 3: Підтвердження

Запитай користувача:
> Виправити всі знайдені проблеми? Або хочете виправити тільки окремі пункти?

## Крок 4: Виправлення

Після підтвердження внеси зміни у файл. Використовуй еталонний файл `bpmn/БП оплати рахунків/оплата-окремого-рахунку.bpmn` як зразок структури.

### Чеклист виправлень (з CLAUDE.md):

**Обов'язкові:**
- [ ] Process ID — латиницею, kebab-case, значення від користувача
- [ ] `isExecutable="true"` на процесі
- [ ] `<zeebe:versionTag value="1.0" />` додано
- [ ] XOR gateway на початку з перевіркою `= is defined(odoo_task_id) and odoo_task_id != null`
- [ ] Default flow на XOR gateway
- [ ] ST_create_main — Service Task з `http-request-smart` для створення головного завдання в Odoo
- [ ] `_id` в body ST_create_main — значення від користувача (НЕ захардкоджене)
- [ ] Swim lanes: мінімум 3 (Система / Виконавець / Керівник)
- [ ] Кожен User Task — `<zeebe:userTask />` + `<zeebe:ioMapping>` для Odoo webhook
- [ ] Кожен User Task має **Task Listener**: Event type = `Creating`, Listener type = `http-request-smart`, **Retries = `3`** (ОБОВ'ЯЗКОВО заповнити Retries, без нього воркер не отримає job)
- [ ] User Task НЕ має одночасно Task Listener І Job worker type — тільки щось одне (інакше дублі)
- [ ] Кожен User Task має 2 boundary events (обидва `cancelActivity="false"`):
  - Нагадування: `<bpmn:timeCycle>R/PT24H</bpmn:timeCycle>`
  - Дедлайн: `<bpmn:timeDuration>P3D</bpmn:timeDuration>`
- [ ] Нагадування → Service Task (http-request-smart) з "⚠️ НАГАДУВАННЯ:" → End
- [ ] Дедлайн → User Task ескалації з "🔴 ЕСКАЛАЦІЯ:" в Lane_manager → End
- [ ] Всі поля Odoo мають префікс `x_studio_camunda_`
- [ ] Всі FEEL-вирази починаються з `=`
- [ ] Назви елементів — українською, ID — латиницею з правильними префіксами (ST_, UT_, GW_, BE_, Flow_, End_)

**Вид задачі (видимість полів):**
- [ ] В body **ST_create_main** (головна задача) додати: `x_studio_camunda_task_type: "full"` — головне завдання показує ВСІ поля
- [ ] В body **кожного User Task** (підзадачі) додати: `x_studio_camunda_task_type: "short"` — підзадачі показують тільки скорочену інформацію
- [ ] Логіка: в Odoo Studio на полях які мають бути тільки в головному завданні → Невидимий: `x_studio_camunda_task_type != "full"`. Це дозволяє керівнику бачити всі дані в головному завданні, а виконавці підзадач бачать тільки те що їм потрібно
- [ ] Значення передаються через body в BPMN, тому впливає тільки на конкретний процес — інші процеси не зачіпає

**Призначення виконавця в body User Task:**
- [ ] По ролі: `x_studio_camunda_role_code: "tech_support"` — автоматизація знайде виконавців з ролі
- [ ] По керівнику: `x_studio_camunda_user_id: x_studio_camunda_manager_user_id` — підставить керівника підрозділу
- [ ] По ID: `x_studio_camunda_user_id: 6` — конкретний користувач
- [ ] УВАГА: використовувати `x_studio_camunda_user_id` (без s), не `x_studio_camunda_user_ids`

**Рекомендовані:**
- [ ] Бізнес-правила в DMN замість FEEL на gateway
- [ ] Call Activity для повторюваних блоків
- [ ] Стрілки без діагональних перетинів

## Крок 5: Результат

Після виправлення покажи користувачу:
1. Короткий список що було змінено
2. Які змінні Odoo потрібно створити (з префіксом `x_studio_camunda_`)
3. Нагадування перевірити Process ID перед деплоєм

Запропонуй запустити auto-layout якщо є скрипт `bpmn_auto_layout.py`.

## Додаткові правила

- Webhook URL — підставляй автоматично залежно від відповіді користувача (демо чи продакшн)
- НЕ використовуй захардкоджений `_id: 252` — завжди значення від користувача
- Назви boundary events — "Нагадування" та "Дедлайн"
- Ескалаційні User Tasks — завжди в Lane_manager
- Service Tasks нагадувань — в Lane_system
