# 🚀 Ship — закомітити, запушити і створити PR

Користувач закінчив роботу над задачею і хоче відправити зміни.

## Крок 1: Перевір статус

```bash
git branch --show-current
git status --short
git log --oneline origin/main..HEAD 2>/dev/null
```

Якщо поточна гілка `main` або `staging` — попередь:
> ⚠️ Ви на гілці `BRANCH`. Спочатку створіть робочу гілку через `/start`.
І заверши.

Збери інформацію:
- `HAS_UNCOMMITTED` — чи є незакомічені зміни (`git status --short`)
- `HAS_UNPUSHED` — чи є закомічені але не запушені коміти (`git log origin/BRANCH..HEAD`)

Якщо немає ні незакомічених змін, ні незапушених комітів — скажи що немає змін для відправки і заверши.

## Крок 2: Визнач що комітити

Якщо `HAS_UNCOMMITTED` — потрібно вирішити що комітити.
Якщо тільки `HAS_UNPUSHED` — переходь одразу до Кроку 4 (пуш).

Отримай список незакомічених файлів:
```bash
git diff --name-only
git diff --name-only --cached
git ls-files --others --exclude-standard
```

### Визначення контексту сесії

Перевір чи є маркер задачі:
```bash
cat .claude-task 2>/dev/null
```

**Є маркер АБО є контекст поточної сесії** (в розмові був `/start` → робота):

Ти знаєш які файли змінювались під час роботи. Комітити потрібно **тільки файли, пов'язані з поточною задачею**:
- Файли, які ти безпосередньо створював або редагував під час роботи
- Весь модуль (папка в `src/custom/`), якщо в ньому були зміни по задачі — навіть якщо частина змін була до сесії
- **НЕ комітити** зміни в модулях/файлах, що НЕ стосуються поточної задачі

Покажи користувачу текстом список файлів які плануєш комітити і запитай через AskUserQuestion:
- "Відправити ці файли?"
- Варіанти: "Так, відправити" / "Скасувати"

**Немає ні маркера, ні контексту** (нова сесія без `/start`):

Покажи список ВСІХ змінених файлів текстом, згрупованих по модулях/папках. Потім запитай через AskUserQuestion:
- "Що з цього потрібно відправити?"
- Варіанти (до 4-х найбільших груп): наприклад "tut_sales (5 файлів)", "tut_web (3 файли)", "Все перераховане", "Скасувати"

Якщо змінених модулів/груп більше ніж 4 — запропонуй "Все перераховане" як одну з опцій, а решту перелічи текстом перед питанням і додай "Other" як можливість вказати конкретні.

Дочекайся відповіді — комітити тільки обрані файли.

Якщо скасовано — зупинись.

## Крок 2.5: Перевірка Odoo модулів (автоматична)

Перед комітом — виконай перевірку змінених модулів (логіка з `/check`).

Визнач які модулі в `src/custom/` торкнуті змінами. Для кожного:

1. **__init__.py**: Перевір що нові .py файли імпортовані
2. **ir.model.access.csv**: Перевір що нові моделі (`_name = '...'`) мають записи доступу
3. **__manifest__.py**: Перевір наявність `version` у форматі `19.0.X.X.X`

Якщо знайдено **блокуючі проблеми** (відсутній імпорт, немає доступу до моделі):
> ⚠️ **Знайдено проблеми в модулях:**
> - `MODULE`: `models/new_file.py` не імпортовано в `__init__.py`
> - `MODULE`: модель `tut.new.model` відсутня в `ir.model.access.csv`

Запитай:
- "Виправити ці проблеми перед комітом?"
- Варіанти: "Так, виправити" / "Ні, ігнорувати"

Якщо "Так" — виправ і додай ці файли до списку на коміт.

Якщо проблем немає — тихо продовжуй далі.

## Крок 3: Коміт

Додай **тільки** відібрані файли:

```bash
git add FILE1 FILE2 ...
```

**НЕ використовуй `git add -A`** — додавай тільки конкретні файли з кроку 2.

Згенеруй зрозуміле повідомлення коміту на основі змін. Формат:

```
[тип] Короткий опис змін

Детальний опис що було зроблено (2-3 рядки).

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

Типи: `feat` (нова фіча), `fix` (виправлення), `refactor`, `style`, `docs`, `chore`.

## Крок 3.5: Синхронізація з main

Перед пушем — переконайся що гілка актуальна відносно main:

```bash
git fetch origin main
```

Спробуй rebase:

```bash
git rebase origin/main
```

Якщо rebase пройшов успішно — продовжуй до Кроку 4.

Якщо rebase впав з конфліктами:

```bash
git rebase --abort
git merge origin/main --no-edit
```

Якщо merge теж впав з конфліктами — покажи:
> ⚠️ Є конфлікти з main. Це означає що хтось змінив ті самі файли.
> Конфліктні файли: (список)

Спробуй автоматично вирішити конфлікти (прийняти обидві версії де можливо).
Якщо не вдається — запитай через AskUserQuestion:
- "Є конфлікти з main які я не можу вирішити автоматично. Що робити?"
- Варіанти: "Покажи конфлікти, я вирішу сам" / "Прийняти версію з main" / "Скасувати ship"

## Крок 4: Пуш

```bash
git push --force-with-lease -u origin BRANCH_NAME
```

## Крок 5: Створи PR

PR завжди створюється як **draft** в **main**.

Перевір чи вже є PR для цієї гілки:
```bash
gh pr list --head BRANCH_NAME --json url --jq '.[0].url'
```

Якщо PR вже існує — скажи що PR оновлено через пуш і покажи посилання. Переходь до кроку 6.

Якщо PR не існує — створи новий:

```bash
gh pr create --title "ЗАГОЛОВОК" --body "BODY" --base main --draft
```

Заголовок PR = повідомлення коміту (без Co-Authored-By).
Body має містити:
- `## Summary` — 1-3 bullet points що зроблено (простою мовою)
- `## Changed modules` — список модулів для оновлення (`-u module1,module2`)
- `## Test plan` — чеклист кроків для тестування
- Підпис: `🤖 Generated with [Claude Code](https://claude.com/claude-code)`

## Крок 5.5: Тригерни Camunda процес (msg_pr_event)

Після створення або оновлення PR — надішли `msg_pr_event` щоб Camunda процес (запущений через `/start`) рушив далі і запустив PR-Agent review.

Отримай номер PR:
```bash
PR_NUMBER=$(gh pr view --json number --jq '.number')
PR_URL=$(gh pr view --json url --jq '.url')
```

Надішли message correlation:
```bash
TOKEN=$(curl -s -X POST "http://10.1.1.74:18080/auth/realms/camunda-platform/protocol/openid-connect/token" \
  -d "grant_type=client_credentials&client_id=orchestration&client_secret=oUV-An_2FED-qYTT" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

curl -s -X POST "http://10.1.1.74:8088/v2/messages/publication" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "msg_pr_event",
    "correlationKey": "BRANCH_NAME",
    "variables": {
      "pr_number": PR_NUMBER,
      "pr_url": "PR_URL",
      "pr_title": "PR_TITLE",
      "pr_author": "tut-ua",
      "head_branch": "BRANCH_NAME",
      "base_branch": "main",
      "repo_full_name": "tut-ua/odoo-enterprise",
      "pr_action": "opened"
    },
    "timeToLive": 3600000
  }'
```

Якщо помилка — повідом але продовжуй (PR-Agent можна запустити вручну пізніше).

## Крок 6: Почекай на review від PR-Agent

Скажи: "⏳ Чекаю на review від PR-Agent (до 3 хвилин)..."

Запам'ятай поточну кількість коментарів. Потім виконуй polling кожні 20 секунд:

```bash
gh pr view --json comments,reviews --jq '.comments | length'
```

Максимум 9 спроб (3 хвилини). Зупинись раніше якщо кількість коментарів збільшилась.

## Крок 7: Проаналізуй та ВИРІШУЙ проблеми

Отримай ВСІ типи коментарів — загальні, review та inline (до конкретних рядків коду):

```bash
gh pr view --json comments,reviews
```

```bash
gh api repos/{owner}/{repo}/pulls/PR_NUMBER/comments
```

Друга команда повертає **inline review comments** — зауваження PR-Agent до конкретних рядків коду. Це найважливіша частина review, саме тут конкретні зауваження.

Об'єднай результати обох команд для повної картини.

### 7.1 Inline зауваження до коду

Якщо PR-Agent залишив **inline зауваження** до конкретних рядків коду:
1. Прочитай кожен файл де є зауваження
2. Зрозумій контекст і суть проблеми
3. **Виправ код** згідно зауважень — не просто повідом, а дійсно виправ
4. Закоміть та запуш виправлення:
```bash
git add FILE1 FILE2 ... && git commit -m "[fix] Виправлення за review PR-Agent

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>"
git push
```
5. Коротко скажи що було виправлено

### 7.2 Merge конфлікти або проблеми з гілками

Якщо PR-Agent або GitHub повідомляє про **merge конфлікти**:
1. **Досліди причину** — визнач які файли конфліктують, прочитай обидві версії, зрозумій ЩО саме конфліктує і ЧОМУ
2. **Поясни ситуацію простою мовою** — хто і коли змінив ці файли, що саме зіткнулось
3. **Запропонуй своє рішення** — покажи конкретно як ти пропонуєш вирішити конфлікт і чому саме так
4. Запитай через AskUserQuestion: "Вирішити конфлікт як я запропонував?" — варіанти: "Так, виконуй" / "Ні, я поясню як"
5. Після підтвердження — вирішуй, комітай і пушай

### 7.3 Загальні зауваження (не inline)

Якщо є загальні зауваження в review (архітектура, безпека, тести):
1. **Досліди кожне зауваження** — прочитай відповідний код, зрозумій контекст
2. Якщо зауваження справедливе і виправлення зрозуміле — **виправ сам**
3. Якщо зауваження потребує рішення від користувача (архітектурне питання, бізнес-логіка) — запитай через AskUserQuestion з конкретними варіантами
4. Якщо зауваження не стосується поточного PR (наприклад, проблеми з іншою гілкою) — поясни це

### 7.4 Немає проблем

Якщо зауважень немає — скажи: "✅ PR-Agent не знайшов проблем."

**ВАЖЛИВО:**
- **НЕ ПРОСТО ПОВІДОМЛЯЙ** про проблеми — **ВИРІШУЙ** їх. Ти асистент розробника, а не репортер.
- Після одного циклу виправлень — СТОП. Не продовжуй автоматично.
- Якщо не можеш щось виправити сам — поясни ЧОМУ і запропонуй конкретні кроки.
